# Vibe Coding Workflow

## Описание (Description)

**Vibe Coding** — это "свободный" режим разработки, фокусирующийся на творческом потоке и интуитивном подходе к написанию кода. В отличие от строгого протокольного подхода, Vibe Coding позволяет разработчику следовать вдохновению, сохраняя при этом структуру и качество.

## Когда использовать (When to Use)

- **Быстрое прототипирование**: Создание MVP, концепт-пруфов
- **Исследовательская разработка**: Поиск новых решений, эксперименты
- **Рефакторинг с вдохновением**: Креативный подход к улучшению кода
- **Локальные утилиты и скрипты**: Небольшие инструменты без строгих требований
- **Обучение и исследование**: Изучение новых технологий и паттернов

## Три Волшебных Пункта (Three Magic Points)

### 1. Создание чек-листа в корне (Create Checklist in Root)

**Цель**: Иметь наглядный план работы, который можно обновлять на лету.

**Действия**:
1. Создайте файл `CHECKLIST.md` в корне проекта
2. Запишите основные задачи и подзадачи
3. Используйте формат:
   ```markdown
   # Checklist для [Название задачи]

   ## Основные задачи
   - [ ] Задача 1
     - [ ] Подзадача 1.1
     - [ ] Подзадача 1.2
   - [ ] Задача 2

   ## Заметки
   - Любые идеи, гипотезы, вопросы
   ```

**Принципы**:
- Чек-лист — живой документ, обновляйте его по мере продвижения
- Не бойтесь менять порядок задач
- Добавляйте новые пункты, если возникают идеи

### 2. Обновление документации (Docs First)

**Цель**: Документация — это не последняя стадия, а часть процесса разработки.

**Действия**:
1. **До написания кода**:
   - Опишите, что должен делать модуль/функция
   - Укажите входные и выходные данные
   - Запишите ключевые сценарии использования

2. **В процессе разработки**:
   - Обновляйте документацию при изменениях
   - Добавляйте примеры использования
   - Описывайте нестандартные решения

3. **После написания кода**:
   - Проверьте, что документация актуальна
   - Добавьте заметки о возможных улучшениях
   - Убедитесь, что документация понятна другим

**Принципы**:
- Документация — это живой код
- Пишите документацию так, как будто объясняете коллеге
- Используйте примеры кода в документации

### 3. Финальная сверка с чек-листом (Final Checklist Review)

**Цель**: Убедиться, что все задачи выполнены и ничего не упущено.

**Действия**:
1. Просмотрите `CHECKLIST.md` и отметьте выполненные пункты
2. Проверьте:
   - Все ли основные задачи выполнены?
   - Есть ли незавершенные подзадачи?
   - Все ли идеи и гипотезы проверены?

3. Проведите саморевью:
   - Код соответствует стандартам проекта?
   - Документация актуальна?
   - Есть ли технический долг?

4. Если остались незавершенные пункты:
   - Оцените их критичность
   - Решите: завершить сейчас, перенести в backlog, или создать отдельную задачу

**Принципы**:
- Честность с собой — не отмечайте выполненным то, что не сделано
- Незавершенные пункты — это не провал, а план на будущее
- Учитесь на каждом незавершенном пункте

## Интеграция с AlfaFlow

### Связь с протоколами

| Аспект | Protocol Mode | Vibe Coding Mode |
|--------|---------------|------------------|
| План | `brief.md` + `plan.md` | `CHECKLIST.md` в корне |
| Документация | Обновляется после | Обновляется в процессе (Docs First) |
| Контроль | Строгий план | Гибкий чек-лист |
| Финал | Code Review + Merge | Self-Review + Checklist Review |

### Когда переключаться

- **Vibe Coding → Protocol**: Если задача становится критической, требует строгого контроля или затрагивает глобальную архитектуру
- **Protocol → Vibe Coding**: Если задача требует креативного подхода, экспериментов или быстрого прототипирования

## Пример использования (Example Usage)

### Сценарий: Создание нового утилитного скрипта

1. **Создание чек-листа**:
   ```markdown
   # Checklist для утилиты экспорта данных

   ## Основные задачи
   - [ ] Создать структуру скрипта
   - [ ] Реализовать чтение данных из БД
   - [ ] Реализовать экспорт в CSV
   - [ ] Добавить логирование
   - [ ] Написать README

   ## Заметки
   - Возможно, стоит добавить поддержку JSON
   - Нужно подумать о прогресс-баре
   ```

2. **Docs First**:
   - Создайте `README.md` с описанием утилиты
   - Опишите аргументы командной строки
   - Приведите примеры использования

3. **Разработка**:
   - Пишите код, обновляя чек-лист и документацию
   - Следуйте вдохновению, но держите чек-лист в виду

4. **Финальная сверка**:
   - Отметьте выполненные пункты
   - Проверьте документацию
   - Примите решение о незавершенных пунктах

## Правила (Rules)

1. **Честность**: Не отмечайте выполненным то, что не сделано
2. **Документация**: Документация — часть процесса, не конец
3. **Гибкость**: Чек-лист можно менять в любой момент
4. **Качество**: Даже в свободном режиме следуйте стандартам кода
5. **Саморевью**: Финальная сверка — обязательный этап

## Ограничения (Limitations)

- Не используйте Vibe Coding для критических компонентов системы
- Не используйте для задач, затрагивающих глобальную архитектуру
- Не используйте для задач с жесткими сроками и требованиями
